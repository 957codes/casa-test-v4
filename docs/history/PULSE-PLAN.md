# Pulse and Relevance Plan

Make Casa recommend what actually matters to the founder, not just what is
structurally eligible. Status: planned 2026-06-27, for review before code.

## Diagnosis

The router ranks next actions by a purely structural score
(`leverage / (slack + 1) x revenue / effort`). It knows what is eligible in the graph,
not what the founder is trying to do. So it surfaced "podcast tour" (a launch playbook)
and "incident response" (an infra loop) for capx, which are eligible but off-priority.
The fix is two halves: capture the founder's pulse, and make the engine think with it.
A bare table also reads as lazy even when the pick is fine, so the output changes too.

## Decisions (2026-06-27)

1. Ranking: weights AND judgment. The pulse sets deterministic priority weights that
   re-rank the eligible set, and the advisor applies final judgment plus reasoning on
   top.
2. Intake: cascade until confident. Adaptive rounds continue until the advisor can fill
   the pulse checklist, bounded by guardrails (below) so it never fatigues.
3. Output: an advisor briefing, not a table. Top one to three actions, each with a why
   tied to the pulse, plus what is being held back and why.

## The pulse (what we capture)

After the project scan and the stage interview, casa-start runs adaptive rounds to fill
a pulse checklist:

- focus: what the founder is actually pushing on now (ranked areas).
- win: what makes the next horizon a win, and the horizon (this week, 30 days, quarter).
- constraint: the single biggest thing in the way.
- anti_priorities: what they are deliberately not doing yet.
- priority_order: the relative ranking across areas.
- risk_appetite and resourcing: solo or team, budget, speed vs safety.
- one_thing: if they could only do one thing this week.

Round protocol (until confident):
- Round 1 asks the reality questions above.
- Each later round is generated from the prior answers and the scan, drilling into what
  the founder raised, and targets only the still-empty checklist slots.
- Stop when the checklist is filled well enough to prioritize, or the founder says build
  it now.

Anti-fatigue guardrails (so "until confident" never drags):
- Every round is skippable ("good enough, build my plan") and the plan builds from
  whatever is filled.
- Hard cap of 5 rounds as a backstop.
- Each round shows the founder how their last answers changed the picture, so the
  questions feel earned, not like a form.
- Quality over quantity: a round asks only what it still needs, not a fixed battery.

## The pulse model in the brain

- `company-brain/pulse.md`: the human-readable pulse (focus, win, constraints,
  anti-priorities, risk, the one thing). Re-read every session.
- `company-brain/pulse.json`: the structured pulse plus the derived priority weights
  the engine consumes:

```json
{
  "focus": ["on-chain token", "agent-company platform"],
  "win": { "definition": "token contract shipped", "horizon": "30d" },
  "constraint": "regulatory clarity",
  "anti_priorities": ["paid marketing", "enterprise sales"],
  "risk_appetite": "ship fast",
  "resourcing": "solo",
  "one_thing": "finish the token contract",
  "weights": {
    "byDepartment": { "Finance": 1.6, "Engineering": 1.3, "Marketing": 0.4, "Sales": 0.3 },
    "byLevel": { "4": 0.4 },
    "promote_ids": ["token-contract"],
    "demote_ids": ["podcast-tour"],
    "default": 1.0
  }
}
```

The advisor interprets the fuzzy pulse into these weights (LLM at the leaf). The router
applies them as plain multiplication (deterministic).

## Pulse to weights to score (kept deterministic)

- Playbooks gain a `department` tag (frontmatter + `_index.json`). This is authored
  once and also replaces the Console's heuristic department derivation. Initial values
  can be generated by the existing keyword heuristic, then reviewed.
- `router.mjs score()` gains a priority-weight factor:
  `score = leverage x (1/(slack+1)) x revenue / effort x priorityWeight(node, weights)`,
  where `priorityWeight = weights.byId? ?? weights.byDepartment[dept]? ?? weights.byLevel[level]? ?? default`,
  with promote_ids and demote_ids as explicit overrides. No pulse means default 1, so
  behavior is unchanged for anyone without a pulse.
- This is a small, testable change: given weights and a node, the weighted score and the
  resulting order are deterministic.

## The advisor layer (judgment + reasoning)

casa-next and casa-priority:
1. Read `pulse.json` (and the rest of the brain).
2. Get the pulse-weighted, eligible candidate set from the engine (router next with
   weights).
3. Apply final judgment: pick the top one to three by founder fit, demote anything still
   off-pulse, never recommend a blocked or ineligible item.
4. Deliver the briefing, not a table:

```
Next: <action>  -- because you said <focus / win>.
Also ready:     <up to 2 more, one line why each>.
Holding back:   <off-pulse item> -- <why not now, tied to anti-priorities or stage>.
```

## Continuous pulse

casa-priority re-checks the pulse each session (a quick "still focused on X, anything
shift this week?") and refreshes `pulse.json` if it changed, so recommendations stay
synced as the founder moves.

## The contract change (rule 4)

Rule 4 says ordering is deterministic. This plan amends it (dated 2026-06-27): the
deterministic core still owns eligibility, dependencies, level gating, the
pulse-weighted score, and all state writes, so it can never recommend an invalid item.
The final relevance ranking and the reasoning move to the advisor over that eligible
set. That is the line that makes Casa feel like a partner instead of a checklist.
